# AI 协作记录

# 一、v0.1

## 背景
本阶段目标是 v0.1：先把项目跑起来并完成“基础渲染”，不做搜索/筛选等后续功能。

---

## 关键对话 1：创建基础项目结构（只搭骨架）
**我的 Prompt（节选）**
- 当前阶段：v0.1
- 子目标：创建基础项目结构
- 需要创建：index.html / style.css / app.js
- 要求：index.html 包含基础结构（header + list 容器），引入 style.css 和 app.js
- 不要实现任何业务逻辑

**AI 回复（节选）**
- 先检查仓库里是否已存在同名文件，避免覆盖
- 创建 index.html（header + ul#spotsList 容器，正确引入 CSS/JS）
- style.css 做基础排版
- app.js 只留空入口（不写业务逻辑）

**我的理解/为什么这样做**
- 先把“能打开页面”的最低版本做出来，后面再逐步加功能，避免一上来改太多东西。

---

## 关键对话 2：读取 spots.json（先能拿到数据，不渲染）
**我的 Prompt（节选）**
- 子目标：在 app.js 中读取 spots.json
- 要求：使用 fetch；成功后打印数据到 console；不实现渲染；只修改 app.js

**AI 回复（节选）**
- 用 fetch('./spots.json') 读取
- res.ok 检查 + try/catch
- 成功后 console.log 输出数据

**我的理解/偏差修正点**
- 我明确要求“只改 app.js、不渲染”，这样可以把任务拆小，出问题更好定位。
- 预览建议用本地静态服务器（Live Server / python http.server），避免 file:// 的限制导致 fetch 失败。

---

## 关键对话 3：实现 renderSpots（按字段渲染 + 格式规则）
**我的 Prompt（节选）**
- 子目标：实现 renderSpots 函数
- 渲染字段：图片、名称、城市、标签、评分、价格、开放时间
- 规则：price=0 → “免费”；rating 保留 1 位小数
- 不要修改 HTML 结构

**AI 回复（节选）**
- 在 app.js 新增 renderSpots(spots)，把内容渲染进现有 ul#spotsList
- rating 用 toFixed(1)，price=0 显示“免费”
- 保持 HTML 不改，只通过 JS 往容器里塞 li

**我的理解/偏差修正点**
- 我强调“不改 HTML 结构”，避免为了渲染去改页面骨架，符合“最小改动”的要求。
- 先把数据能展示出来，后续再做样式优化。

---

## 关键对话 4：卡片样式优化（grid + 图片自适应）
**我的 Prompt（节选）**
- 子目标：优化卡片样式
- 要求：使用 grid 布局；图片自适应；卡片整齐排列；只修改 style.css

**AI 回复（节选）**
- .spots-list 改为 grid，自动换列
- li 作为卡片
- img 用 width:100% + aspect-ratio + object-fit:cover 保持整齐

**我的理解**
- 把结构（HTML）、逻辑（JS）、样式（CSS）分开逐步做，修改范围更可控。

---

## 小结（我这阶段学到的）
- 和 AI 协作时，把需求写成“阶段 + 子目标 + 只改哪些文件 + 不要做什么”，会更省时间、也更不容易跑偏。
- 每次只做一个小目标，能快速看到效果，也方便用 git 记录。

---

# 二、v0.2

## 背景
本阶段目标是 v0.2：在 v0.1 的“能渲染”基础上，加上“搜索”体验，但依然按规则：小步迭代、不要提前做筛选/扩展。

---

## 关键对话 1：只加搜索框（先做 UI，不写逻辑）
**我的 Prompt（节选）**
- 当前阶段：v0.2
- 子目标：在 index.html 添加搜索框
- 要求：placeholder: 搜索景点名称/城市；不实现逻辑

**AI 回复（节选）**
- 只在 `index.html` 增加一个 `input type="search"`，设置 placeholder
- 不写任何 JS 搜索逻辑

**我的理解/偏差修正点**
- 我把“先有输入框”当成一个独立小任务，避免 UI 和逻辑混在一起改太多。

---

## 关键对话 2：目录结构变化导致读取失败（修正 fetch 路径 + 明确规范结构）
**我的 Prompt（节选）**
- 修改：由于文件夹结构改变，现在无法读取 json 文件，修改，并为我生成一个规范的文件夹结构

**AI 回复（节选）**
- 先找清楚 `spots.json` 与 `src/app.js` 的相对位置
- 把 `fetch("./spots.json")` 修正为能读到 JSON 的路径（从 `src/` 读取上层的 `spots.json`）
- 给出更清晰的目录建议：`spots.json` 和 `img/` 放根目录，`src/` 放 `index.html/style.css/app.js`

**我的理解/偏差修正点**
- 这里出现了“环境/目录变化”带来的偏差：逻辑本身没错，但路径不对就会 404。
- 先把结构定下来，再继续做 v0.2 功能，不然每一步都可能因为路径问题反复踩坑。

---

## 关键对话 3：打开网站出现目录列表页（补根目录入口）
**我的 Prompt（节选）**
- 现在把地址切换到了项目根目录，但是一打开网站如图所示（目录列表页），不对，修改

**AI 回复（节选）**
- 解释原因：服务器根目录缺少 `index.html`，所以显示“目录索引”
- 用最小改动补一个根目录 `index.html` 作为入口，引入 `./src/style.css` 和 `./src/app.js`

**我的理解/偏差修正点**
- 这也是“上下文选择”的问题：服务器默认找根目录 `index.html`，所以补入口比改服务器配置更简单、更稳。

---

---

# 三、v0.3

## 背景
本阶段目标是 v0.3：在已有“搜索”的基础上增加“标签筛选/过滤”，并要求支持多选、与搜索联动（取交集 AND），再补上重置能力和更好的样式。

---

## 关键对话 1：生成标签按钮（从 JSON 提取 tags 并去重）
**我的 Prompt（节选）**
- 当前阶段：v0.3
- 子目标：生成标签按钮
- 要求：从 JSON 提取 tags、去重、渲染为按钮
- 只修改 app.js 和 index.html

**AI 回复（节选）**
- 在 `index.html` 增加标签区域容器（如 `#tagFilters`）
- 在 `app.js` 读取到数据后提取所有 tags，用 Set 去重
- 把去重后的 tags 渲染成一排 `<button>`

**我的理解**
- 先把“标签 UI 能显示出来”作为独立小任务，后面再接多选、筛选逻辑，避免一次性做太多。

---

## 关键对话 2：实现标签多选（只做选中状态，不动搜索逻辑）
**我的 Prompt（节选）**
- 子目标：实现标签多选
- 要求：点击切换选中状态；支持多个标签同时选中
- 不要修改搜索逻辑

**AI 回复（节选）**
- 用 `Set` 保存已选标签
- 通过点击事件切换标签是否选中（支持多选）
- 不改搜索相关函数

**我的理解/偏差修正点**
- 我强调“不改搜索逻辑”，把功能边界分清楚：搜索先稳定，再叠加筛选。

---

## 关键对话 3：点击 tags 页面没反应（排查并最小代价修复）
**我的 Prompt（节选）**
- 问题：点击 tags 后页面没有反应
- 要求：找到问题并用最小代价修改；并修改样式区分选中/未选中

**AI 回复（节选）**
- 原因：JS 其实切换了状态，但没有 CSS 体现，用户就感觉“没反应”
- 最小修复：点击时直接切换按钮的 `is-selected` / `aria-pressed`
- 在 CSS 增加选中态样式（颜色/边框/背景）区分

**我的理解/偏差修正点**
- “无反应”不一定是 JS 不执行，也可能只是 UI 没视觉反馈；先定位原因再改，避免把代码改复杂。

---

## 关键对话 4：实现标签筛选 + 与搜索联动（AND）
**我的 Prompt（节选）**
- 要求：根据选中的标签过滤景点数据
- 要求：实现搜索 + 标签筛选联动；同时满足：搜索 AND 标签
- 修改：filterSpots 函数

**AI 回复（节选）**
- 增加/整理 `filterSpots`：把“搜索条件 + 标签条件”的 AND 逻辑集中到一个函数里
- 搜索：模糊匹配 name/city；标签：多选命中任意一个（OR），但整体与搜索做 AND
- 点击 tag 后触发一次列表刷新，让当前搜索词和标签一起生效

**我的理解**
- 把过滤规则放到 `filterSpots`，后面要改规则（交集/并集、精确匹配等）也更集中、更好维护。

---

## 关键对话 5：实现重置功能（清空搜索 + 清空标签 + 恢复全部）
**我的 Prompt（节选）**
- 子目标：实现重置功能
- 要求：清空搜索、清空标签、恢复全部数据

**AI 回复（节选）**
- 在页面增加一个重置按钮（如 `#resetBtn`）
- 点击后清空输入框、清空 `selectedTags`、重新渲染全量列表与按钮状态

**我的理解**
- 重置是 v0.3 的“兜底操作”，能显著提升可用性，也方便验收。

---

## 关键对话 6：样式对齐参考图（卡片与标签更像示例）
**我的 Prompt（节选）**
- 子目标：修改景点样式，学习样式图

**AI 回复（节选）**
- 最小改动方式：在 `renderSpots` 给关键元素加 class，并把 tags 渲染成多个胶囊标签
- CSS 用 grid 排版：标题左/城市右、标签胶囊、评分/价格信息块、卡片浅色阴影

**我的理解**
- 先让信息层级和布局接近示例图（结构清晰），再做颜色/细节微调，效果更稳定。

---

## 小结（v0.3）
- 我更习惯先做“看得见的 UI”，再接“可用的逻辑”，最后补“体验与样式”，每一步都能快速验证。
- 遇到“没反应/不对劲”的问题，先判断是路径/数据/事件还是纯样式反馈，再做最小修复，减少返工。
